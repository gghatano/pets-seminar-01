仮名加工情報・匿名加工情報 事例再現デモプロジェクト 設計・実装指示書

1. プロジェクトの目的

個人情報保護委員会事務局が公開している、仮名加工情報・匿名加工情報に関する事例レポートを参照し、掲載されている事例のデータ加工をPythonで再現するデモコンテンツを作成する。

最終成果物はGitHub上で公開・管理し、Jupyter NotebookはGoogle Colaboratoryから実行可能なデモ・教材として提供する。

本プロジェクトでは、単にレポート記載の加工処理をPythonコードとして実装することを目的としない。

実際のデータ加工案件を想定し、

データ構造の把握
↓
列ごとの情報特性・識別性・機微度の評価
↓
利用目的の整理
↓
利用目的上必要なデータと粒度の整理
↓
列ごとの加工方針の決定
↓
具体的な加工仕様の策定
↓
Pythonによる加工の実行
↓
加工前後の比較
↓
加工後データの利用

という一連のプロセスを、ダミーデータとNotebookを通じて追体験できる教材を作成する。

初期フェーズでは、以下の2事例を作成する。

1. 仮名加工情報の事例：1件
2. 匿名加工情報の事例：1件

まず、仮名加工情報の事例を対象として設計・実装する。

2. 基本方針

2.1 一次情報

個人情報保護委員会事務局が公開している対象レポートを一次情報として使用する。

対象事例について、必ず一次情報を確認した上で設計すること。

レポートに記載されている内容と、本プロジェクトで教材作成のために独自に補完した内容を明確に区別する。

レポートに記載されていない加工方法や法的要件を、レポート記載事項であるかのように扱わない。

⸻

2.2 教材としての位置づけ

本Notebookは、

* 仮名加工情報・匿名加工情報の制度理解
* データ加工設計プロセスの理解
* Pythonによる具体的な加工方法の理解

を目的とする。

実際の個人データに対する法令適合性を保証するものではないことを明記する。

7. 事例整理

対象レポートを確認し、以下を整理する。

7.1 事例背景

* 事業者
* 事業内容
* 保有データ
* 従来の利用目的
* 新たな利用目的
* 仮名加工情報を作成する背景
* 加工後の利用方法

7.2 元データ

レポートに記載されているデータ項目を可能な限り網羅的に抽出する。

各項目について、

* 項目名
* 項目の意味
* 想定データ型
* データ例
* 1人につき1つ存在する情報か
* 1人につき複数存在する履歴情報か
* 他テーブルとの結合に利用される情報か

を整理する。

7.3 加工内容

レポート上の加工方法を項目単位で整理する。

少なくとも以下に分類する。

* 削除
* 置換
* 一般化
* 丸め・粒度変更
* 加工なし
* その他

各加工について、レポートから読み取れる範囲で加工理由を整理する。

8. ダミーデータ仕様書

ダミーデータは単なるランダムデータではなく、対象事例の加工設計を再現するために必要な特徴を持たせる。

8.1 データモデル

対象事例の元データ構造を分析し、適切なデータモデルを設計する。

例えば、

* 顧客マスタ
* Webアクセス履歴
* 購買履歴

などへの分割を検討する。

レポート上の構造を尊重しつつ、実際の業務データとして自然な構造を選択する。

以下を定義する。

* テーブル一覧
* 各テーブルの役割
* 主キー
* 外部キー
* テーブル間のリレーション
* 1レコードの粒度

単一テーブルに非正規化する場合は、その理由を説明する。

8.2 データ件数

以下を定義する。

* 顧客数
* 顧客1人当たりの購買件数
* 顧客1人当たりのアクセス件数
* 各テーブルの総レコード数

Notebookを短時間で実行できる規模とする。

8.3 データ分布

完全な一様乱数にはしない。

加工後に利用目的に対応する簡単な分析が成立するよう、人工的な傾向を持たせる。

例えば、

* 地域による顧客数の偏り
* 年代による購買商品の違い
* 地域による購買金額の違い
* 商品カテゴリと顧客属性の関係
* Webアクセスと購買の関係

などを検討する。

ただし、レポートに記載されていない設定については、教材用に人工的に設定したものであることを明示する。

8.4 同一人物の履歴

同一顧客について複数の購買履歴・アクセス履歴を生成する。

加工後も、利用目的上必要な範囲で同一人物の履歴を分析できる構造とする。

8.5 再現性

乱数seedを固定する。

原則として以下を中心に使用する。

* Python標準ライブラリ
* pandas
* numpy

外部ライブラリを利用する場合は、その必要性を説明する。

10. 列ごとの情報特性・機微度分類

加工方法を決定する前に、各列の情報特性を評価する。

以下の観点を使用する。

10.1 直接識別性

その情報単独で本人を直接識別し得るか。

10.2 間接識別性

他の情報との組合せによって本人の識別につながる可能性があるか。

10.3 機微度

本人が特定された場合のプライバシー上の影響等を考慮し、必要に応じて、

* 低
* 中
* 高

等で分類する。

この分類は法令上の「要配慮個人情報」等の分類とは異なる、本教材における加工設計上の補助指標であることを明示する。

10.4 情報分類

必要に応じて以下に分類する。

* 直接識別子
* 準識別子
* 属性情報
* 行動・履歴情報
* 内部管理識別子
* その他

用語を使用する場合は定義を記載する。

10.5 結合キー性

複数テーブルや複数履歴を結合するために必要な情報かを評価する。

10.6 分類結果

以下の形式で整理する。

項目	情報分類	直接識別性	間接識別性	機微度	結合キー性	備考

⸻

11. 利用目的と必要情報の整理

加工方針を決定する前に、利用目的から必要な情報を整理する。

以下の形式を基本とする。

分析・利用目的	必要な情報	必要な粒度	備考

例えば、

地域別の分析に住所情報が必要であっても、詳細住所まで必要なのか、市区町村単位で十分なのかを検討する。

この整理により、

利用目的
↓
必要情報
↓
必要粒度

を明確にする。

⸻

12. 加工設計書

加工方針は機微度のみで決定しない。

以下の複数軸を踏まえて判断する。

識別性 × 機微度 × 利用目的上の必要性 × 必要粒度

例えば、

* 直接識別性が高く、分析に不要 → 削除を検討
* 元の識別子は不要だが個人単位の履歴結合が必要 → 整理番号等への置換を検討
* 間接識別性があり、詳細値は不要 → 必要粒度まで一般化
* 行動・履歴情報で利用目的に直接必要 → 必要性を確認した上で維持

という判断を行う。

以下の形式で整理する。

項目	情報分類	識別性	機微度	分析上の必要性	必要粒度	加工方針	理由

各加工方針について、可能な範囲で以下を区別する。

* レポートに明示されている加工
* レポートから合理的に読み取れる加工
* 教材作成のために独自に設定した加工

⸻

13. 加工仕様書

加工設計書で決定した方針を、Pythonで実装可能な具体的処理に落とし込む。

以下の形式で整理する。

No.	元テーブル	元項目	加工方針	加工処理	加工後項目	データ型	Python実装方針	根拠

例えば以下を具体化する。

* カラム削除
* 整理番号の付与
* 識別子の置換
* 生年月日から年齢への変換
* 年齢から10歳単位の年代への一般化
* 詳細住所から市区町村への一般化
* 履歴項目の維持

加工後のスキーマも定義する。

⸻

14. 加工後テーブル定義書

加工後データについて、加工前と同じ形式でテーブル定義書を作成する。

さらに以下の対応表を作成する。

加工前項目	加工後項目	処理
元識別子	整理番号	置換
氏名	―	削除
生年月日	年代	一般化
詳細住所	市区町村	一般化

加工によるスキーマ変化を追跡可能にする。

15. ダミーデータ生成

16. Notebookの位置づけ

Notebookはデモコンテンツである。

Notebook内で以下は原則として実施しない。

* ダミーデータ生成
* テーブル定義書の生成
* 加工設計書の生成

これらはGitHub上のdocsおよびsrcを正本とする。

Notebookは、

事例を理解する
↓
データを見る
↓
加工判断を理解する
↓
実際に加工する
↓
結果を見る
↓
加工後データを利用する

というデモ体験に集中する。

6. 利用目的と必要情報

今回の利用目的と、そのために必要な情報・粒度を簡潔に説明する。

7. 加工方針

主要項目について以下を表示する。

項目	分析上の必要性	必要粒度	加工方針

8. 加工処理

Pythonで加工を実行する。

加工の中核処理は外部モジュールに隠蔽しすぎず、Notebook上で処理内容を確認できるようにする。

理解しやすい単位で段階的に実行する。

例：

1. 識別子の処理
2. 直接識別情報の削除
3. 生年月日等の一般化
4. 住所等の一般化
5. 不要項目の削除
6. 分析に必要な履歴情報の維持

各ステップで、

* 何を加工するか
* なぜ加工するか
* Pythonコード
* 加工結果

を確認できるようにする。

9. 加工前後比較

以下を比較する。

* カラム構成
* サンプルレコード
* 削除された情報
* 一般化された情報
* 維持された情報

「加工によって何が失われ、何が維持されたか」が理解できるようにする。

10. 加工後データによる分析

加工後データを用いて、対象事例の利用目的に対応する簡単な分析を実施する。

高度な機械学習は不要とする。

pandasのgroupby等を中心に、

「加工後も目的とする分析が可能である」

ことを確認する。

11. 確認テスト

assert等を利用し、加工仕様どおりに処理されたことを確認する。

例えば、

* 削除対象カラムが存在しない
* 必要な加工後カラムが存在する
* レコード数が意図せず変化していない
* 同一人物の履歴関係が想定どおり維持されている

ことを確認する。

18. Notebookと設計書の関係

詳細仕様はGitHub上のdocs配下を正本とする。

Notebookにはデモ理解に必要な情報のみ要約して掲載する。

例えば、詳細なテーブル定義書をNotebook内に全文掲載する必要はない。

一方で、Notebookから外部資料へのリンクだけを提示し、Notebook単体では加工判断が理解できない構成にもしてはならない。

原則として、

詳細はGitHubの設計書

理解に必要な要点はNotebook

という役割分担とする。

⸻

19. Google Colaboratory対応

NotebookはGoogle Colaboratoryで追加設定なく実行できることを目標とする。

以下を満たすこと。

* ローカルファイルに依存しない
* GitHub上の公開データを直接取得する
* 特殊なパッケージへの依存を可能な限り避ける
* 必要なライブラリをNotebook冒頭で明示する
* セルを上から順番に実行すれば完了する
* 実行時間を過度に長くしない

README.mdには各Notebookへの「Open in Colab」導線を設けることを想定する。

⸻

20. Notebook実装上の原則

Notebookは教材としての可読性を優先する。

単純なカラム削除等まで過度に関数化しない。

加工処理の中核部分は、Notebook上でコードを読めば何をしているか理解できるようにする。

一方で、住所の一般化等、一定の処理ロジックが必要なものは関数化してよい。

ただし、関数の内容はNotebook上で確認可能とする。

Notebookは「実用的な加工ライブラリ」の実装を目的とせず、「加工設計と加工処理を理解するデモ」であることを優先する。

23. 品質基準

23.1 再現性

事務局レポートの対象事例と加工内容が対応していること。

23.2 トレーサビリティ

24. 最終的に目指す利用体験

利用者はGitHubのREADMEから対象事例を選択し、「Open in Colab」からNotebookを開く。

Notebookを上から順番に読み、実行することで、

1. どのような事例なのかを理解する
2. どのようなデータが存在するかを見る
3. 各データ項目がどのような性質を持つかを理解する
4. 利用目的から必要な情報を考える
5. なぜ各列を削除・置換・一般化・維持するのかを理解する
6. Pythonで実際に加工する
7. 加工前後の違いを確認する
8. 加工後データでも目的とする分析ができることを確認する

という一連の体験ができることを目標とする。

本プロジェクトの中心的な価値は、加工コードそのものではなく、

「データを見て、利用目的を考え、情報特性を評価し、適切な加工方針を設計し、その結果を実装・確認する」

というデータ加工設計の思考プロセスを、具体的な事例を通じて理解できることに置く。

この版をプロジェクト全体の spec.md 相当として使い、エージェントにまず case01の設計8文書を作成 → 整合性レビュー → ダミーデータ生成 → Notebook実装 の順で進めさせる構成が適しています。

⸻

25. 配信・公開構成（GitHub Pages / Colab の3層構成）

教材の「思考プロセス」を第三者に見せるための配信構成を定める。媒体を役割で分ける。

25.1 3層構成

| 層 | 媒体 | 内容 | 対応する設計成果物 |
|----|------|------|--------------------|
| ① 検討 | GitHub Pages（HTML） | 事例理解 → ダミーデータ・テーブル定義 → 利用目的 → 情報特性評価 → 加工設計の決定 | 01, 02, 03, 04, 05, 06 |
| ② 実装 | Google Colaboratory（Notebook） | 加工設計にもとづく加工処理の実行 | 06 の実装、notebooks/ |
| ③ 結果 | GitHub Pages（HTML） | 加工前後の比較・確認テスト結果・加工後分析 | 07, 09_results |

25.2 原則

* HTML サイトは `docs/` 配下の Markdown を正本として生成し、内容を二重管理しない（第18節と同じ役割分担を配信にも適用する）。
* ③の「結果・チェック結果」は、②の Notebook を実行した出力を HTML 化して提示する。加工ロジックを HTML と Notebook に二重実装しない。Notebook を唯一の実行ソースとし、「サイト上の結果 = 実際の実行結果」を保証する。
* GitHub Pages は公開だが、扱うのは合成データと教材ナラティブのみとする。

25.3 実装

* サイト生成: MkDocs + Material テーマ。Notebook 埋め込みは `mkdocs-jupyter`。
* 設定は `mkdocs.yml`。ローカルは `uv run mkdocs serve`、ビルドは `uv run mkdocs build --strict`。
* 公開は GitHub Actions（`.github/workflows/pages.yml`）で `main` への push を契機に自動デプロイする。
* ②の Notebook は正本を `notebooks/` に置き、Colab から実行する。③に実行済み結果を埋め込む場合は、ビルド時に `notebooks/` の Notebook をサイトへ取り込む。
