## 2. レポート事例から抽出した要件（R1 事例1）

### 2.1 事例背景〔報告〕
- 事業者: 食品のオンライン通信販売事業を行う **事業者A**
- 新規事業: 特定産地で有機農法により収穫された食品を、首都圏の**実店舗**にて販売する事業（実店舗事業）
- 保有データ: オンライン通信販売事業に関連して取得した顧客の個人情報（後述の元データ）
- 従来の利用目的（図表1-1）:
  - お客様がご購入された商品を提供するため
  - お客様が閲覧した当社ウェブページの履歴に基づき、お客様の興味に応じた新商品に関する広告を行うため
- 新たな利用目的（＝仮名加工情報を作成する背景）: オンライン通販で取得した個人情報を分析し、ある地域でどのような顧客層（年齢・性別）がどのような商品に関心を有しているかを分析し、実店舗事業の**出店計画**を検討する（従来の利用目的には含まれない）
- 加工後の利用目的（図表1-5、変更後）: 「お客様の属性（年代・性別・居住地）、商品購入履歴、当社ウェブページの閲覧履歴から、各地域のお客様の興味やニーズの傾向を分析し、実店舗事業の計画を行うため」
- 分析に必要な粒度: 住所=**市区町村単位**（実店舗の商圏判定）、年齢=**10歳区切りの年代**。購入年月日・購入品目はできる限り加工せずに分析したい。

### 2.2 元データ項目（図表1-1）〔報告〕

**個人属性情報**: 会員ID / 氏名 / 生年月日 / 性別 / 郵便番号 / 住所 / 携帯電話番号 / 電子メールアドレス / クレジットカード番号

**履歴情報（購買に関する情報）**: Cookie ID / 自社ウェブページのアクセス履歴 / 購入年月日 / 購入品目 / 購入数量 / 購入金額

> 図表1-2 でデータセットを「個人属性情報」と「履歴情報」に分類している。〔報告〕

### 2.3 レポート上の加工方法（図表1-3）〔報告〕

| 項目 | 分類 | 本事例での加工方法 | 加工の型 |
|------|------|--------------------|----------|
| 会員ID | 属性 | 分析に不要＋事業者内の識別禁止義務抵触リスク低減のため削除し、**整理番号**を付す | 置換 |
| 氏名 | 属性 | 全部削除 | 削除 |
| 生年月日 | 属性 | 年齢に換算し「10代」〜「80代以上」に置き換え（10歳区切り） | 一般化（丸め） |
| 性別 | 属性 | 生年月日・住所を加工しているため加工しない | 加工なし |
| 郵便番号 | 属性 | 住所を加工したものを利用するため不要。削除 | 削除 |
| 住所 | 属性 | 市区町村名より細かい情報を削除（市区町村名までに丸め） | 一般化（丸め） |
| 携帯電話番号 | 属性 | 本人到達性・共用性のため削除 | 削除 |
| 電子メールアドレス | 属性 | 本人到達性・共用性のため削除（施行規則第31条第1号該当時は削除義務）〔法令〕 | 削除 |
| クレジットカード番号 | 属性 | 施行規則第31条第3号に基づき削除〔法令〕 | 削除 |
| Cookie ID | 履歴 | 本人到達性のため削除（会員IDをレコード識別子に使用、分析に不要） | 削除 |
| アクセス履歴 | 履歴 | キャンペーン反応・閲覧→購入の分析ニーズがあり加工しない | 加工なし |
| 購入年月日 | 履歴 | 加工しない | 加工なし |
| 購入品目・購入数量・購入金額 | 履歴 | 加工しない | 加工なし |

> 加工の考え方: 仮名加工情報には匿名加工情報のような一律の加工基準がなく、識別禁止義務・削除情報等の安全管理等を前提に、個人単位の特徴を残した分析が可能。識別性×利用目的上の必要性×必要粒度で項目ごとに判断している。〔報告〕

## 3. 制度上の要件（仮名加工情報 / 確認済みの論点）

R1・ガイドライン（R2）由来。事例事実と混同しないよう〔法令〕で明示する。

- 施行規則第31条に基づく加工（R1 事例1で言及）: 第1号（特定の個人を識別できる記述等の削除／置換）、第3号（不正利用により財産的被害が生じるおそれのある記述等＝クレジットカード番号全桁の削除）等。〔法令〕
- 作成後の義務: 識別行為の禁止（作成元の個人情報との照合による本人識別の禁止）、削除情報等の安全管理措置、本人への連絡等の禁止 等。〔法令〕
  - 本事例で会員IDを整理番号に置換するのは、事業者内で仮名加工情報と作成元個人情報を照合し**識別禁止義務**に抵触する可能性を低減するため。〔報告〕
- 詳細条文・要件はガイドライン（[R2](../references/README.md)）で最終確認する。

> 本教材は制度理解の補助であり、実データへの法令適合性を保証しない（spec.md §2.2）。本教材の「機微度」分類は法令上の要配慮個人情報とは別概念（spec.md §10.3）。

## 4. 教材要件（spec.md 由来）〔独自/簡略〕

- データ加工設計の**思考プロセス**（データ理解→情報特性評価→利用目的→加工設計→実装→確認）を追体験できること（spec.md §24）。
- 3層構成で提示: ① 検討=HTML / ② 実装=Colab / ③ 結果=HTML（spec.md §25）。
- レポート記載・法令・独自設計・簡略化を区別（spec.md §23.5）。
- 完結性: 設計成果物だけで別の実装者が Notebook を実装できること（spec.md §23.3）。

## 5. データ要件（ダミーデータが満たすべき条件）〔独自〕

R1 事例1のデータ構造を再現するダミーデータを作る。詳細は [02_dummy_data_spec.md](02_dummy_data_spec.md)。

- データモデル: 図表1-1/1-2 の構造を尊重し、個人属性（顧客マスタ）と履歴情報（購買履歴・Webアクセス履歴）に分割する方向で検討。会員IDで結合。
- 分布: 一様乱数にせず、加工後に「地域×年代・性別×購入商品」の傾向分析が成立するよう人工的傾向を持たせる（レポート未記載の設定は〔独自〕と明示）。
- 同一顧客に複数の購買・アクセス履歴。乱数 seed 固定で再現可能。
- 首都圏を想定した市区町村・年代分布など、実店舗出店分析が体験できる規模・傾向。
- Colab で短時間に実行できる規模。
